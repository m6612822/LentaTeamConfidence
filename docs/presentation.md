# Презентация ShelfScan — материалы и спикер-скрипт

Три согласованных артефакта:
- **`presentation.html`** — готовая презентация (открыть в браузере,
  стрелки/пробел листают, `F` — полноэкранный). Самодостаточна, без
  интернета. 11 слайдов, минимализм, тёмная тема.
- **`presentation_prompt.md`** — промпт-бриф для пересборки/редизайна
  (Gamma / Marp / дизайнер).
- **этот файл** — что говорить на каждом слайде.

Перед защитой заполнить плейсхолдеры `<…>` в `presentation.html`
(команда, имена/контакты, git URL, HF Space URL) — они подсвечены жёлтым.

## Спикер-скрипт (≈5–6 мин)

**1. Титул.** «ShelfScan — превращаем видео робота с полки в
структурированные данные о ценниках. Команда ___.»

**2. Команда.** Коротко: кто за что отвечал.

**3. Задача.** «Вход — видео движущегося робота. Выход — CSV, строка на
ценник, 29 полей. Метрика — доля ценников с точностью ≥80%.»

**4. Анализ данных.** «Сразу нашли три вещи, определившие подход. Первое —
видеопоток со смещением start_time, кадр 0 ≈ 824 мс; без фикса вся
разметка не совпадает. Второе — штрихкоды и QR в этом сжатии физически не
декодируются: проверили полным сканом, zxing 0 из 412 кадров, нейро
super-res WeChat 0 из 275. Третье — OCR деградирован сжатием и блюром.»

**5. Архитектура.** «Пайплайн полностью локальный, permissive-лицензии,
ONNX/CPU, готов к RKNN. Ядро — multi-frame fusion (выделить голосом).»

**6. Fusion.** «Один кадр нечитаем — на нём цена даёт мусор `149`. Но
робот видит ценник десятки раз: трекаем тег по проходу, субпиксельно
выравниваем по ECC, медианно сливаем, прогоняем нейро Real-ESRGAN ×4 —
и из мусора получаем читаемое: бренд, объём, цену.»

**7. Доказательство (главный слайд).** «Мы строго доказали: логика верна.
При идеальном OCR наша схема даёт 97.8% ценников с точностью ≥80% —
расчёт по 274 размеченным строкам. На выданном же видео — 0 из 57, и при
строгой, и при посимвольной оценке. Вывод: предел задаёт качество
входного видео, а не наше решение. Это воспроизводимый научный результат,
а не оправдание.»

**8. Прозрачность.** «ТЗ требует прозрачности. QR оптически мёртв —
поэтому QR-поля мы выводим из визуальных по эквивалентности, выученной
из самих данных: price1=price_default 97%, price4=price_card 97%,
остальные почти всегда „нет“. Это автостатистика, не ручная разметка —
и мы это честно раскрываем.»

**9. Демо.** Показать Gradio UI вживую/скрином: видео → таблица → CSV.

**10. Ограничения и масштаб.** «На несжатом или edge-RAW потоке те же
компоненты идут к 97.8%. Модели — ONNX, конвертируются в RKNN int8;
конфиг data-driven под новые форматы ценников; детектор дообучается на
новых полках по подходу с provided-GT.»

**11. Ссылки.** Репозиторий и задеплоенное демо.

### Ответы на острые вопросы (Q&A)
- *Почему метрика 0?* — Информация уничтожена сжатием выданного видео;
  доказали, что её не вытягивают ни zxing, ни нейро-SR. Архитектура
  доказуемо даёт 97.8% на нормальном входе.
- *Платные/облачные сервисы?* — Запрещены ТЗ; и они не вернут
  уничтоженные сжатием штрихкоды. Узкое место — данные, не бюджет.
- *Ручная разметка?* — Нет. Детектор — на provided-GT организаторов
  (не участников) + авторазметка Grounding-DINO; конфиг калиброван
  статистикой данных.
